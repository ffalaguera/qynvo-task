# Qynvo Task — Travel Tips API

A Laravel API endpoint that generates personalised travel tips using the Anthropic Claude API. Built as a technical task for Qynvo.

## How It Works

The API accepts a traveller's itinerary (destination, dates, and activities) and returns 3 personalised travel tips generated by Claude AI. Results are cached to avoid redundant API calls.

## Tech Stack

- **PHP 8.x** / **Laravel 13**
- **Anthropic Claude API** (claude-sonnet-4-6)
- **SQLite** (default Laravel database)
- **Laravel Cache** (file driver) for response caching

## Project Structure

app/
├── Http/
│   ├── Controllers/
│   │   └── TravelTipsController.php   # Receives request, returns JSON response
│   └── Requests/
│       └── ItineraryRequest.php       # Validates input before reaching controller
└── Services/
└── ClaudeService.php              # All Claude API logic: prompt, HTTP call, parsing, cache

## Setup

```bash
# 1. Clone the repository
git clone https://github.com/ffalaguera/qynvo-task.git
cd qynvo-task

# 2. Install dependencies
composer install

# 3. Configure environment
cp .env.example .env
php artisan key:generate

# 4. Add your Anthropic API key to .env
ANTHROPIC_API_KEY=your-key-here

# 5. Run migrations
php artisan migrate

# 6. Start the server
php artisan serve
```

## API Usage

### POST /api/travel-tips

**Request:**

```json
{
    "destination": "Tokyo",
    "start_date": "2026-07-15",
    "end_date": "2026-07-20",
    "activities": ["visit temples", "try sushi", "explore Shibuya"]
}
```

**Successful Response (200):**

```json
{
    "success": true,
    "data": [
        {
            "title": "Beat the Crowds at Senso-ji Temple",
            "description": "July is peak tourist season in Tokyo, so arrive at Senso-ji before 7am..."
        },
        {
            "title": "Navigate Tsukiji Outer Market for the Freshest Sushi",
            "description": "Head to Tsukiji Outer Market early morning around 8-9am..."
        },
        {
            "title": "Time Your Shibuya Crossing Visit Strategically",
            "description": "Plan your Shibuya Crossing visit for evening hours around 8-10pm..."
        }
    ],
    "cached": false
}
```

**Validation Error (422):**

```json
{
    "success": false,
    "message": "Validation failed.",
    "errors": {
        "destination": ["The destination field is required."]
    }
}
```

**Server Error (500):**

```json
{
    "success": false,
    "message": "Failed to generate travel tips.",
    "error": "Claude API returned status: 401"
}
```

## Design Decisions

### Separation of Concerns

The code is split into three layers, each with a single responsibility:

- **ItineraryRequest** handles validation only. If input is invalid, the request never reaches the controller. This keeps validation logic isolated and reusable.
- **TravelTipsController** only connects the request to the service and formats the response. It contains zero business logic.
- **ClaudeService** encapsulates all Claude API interaction: prompt building, HTTP calls, response parsing, and caching. If we switch from Claude to another LLM provider, only this file changes.

This mirrors the principle I applied during my internship at DISID, where we used hexagonal architecture to keep business logic decoupled from infrastructure.

### Error Handling Strategy

Errors are handled at multiple levels:

1. **Validation layer** (ItineraryRequest): Returns 422 with specific field errors before any processing.
2. **Configuration check** (ClaudeService): Throws immediately if API key is missing, with a clear message.
3. **API call** (ClaudeService): Catches timeouts, network errors, and non-200 responses with descriptive messages.
4. **Response parsing** (ClaudeService): Validates that Claude returned valid JSON with the expected structure.
5. **Controller** (TravelTipsController): Catches any exception and returns a clean 500 JSON response.

The API never returns raw Laravel error pages — every error is a structured JSON response.

### Caching

Results are cached using Laravel's Cache facade with a key derived from the MD5 hash of the full itinerary. The same itinerary always produces the same cache key. Cache duration is 24 hours.

This means:
- Identical requests don't consume API quota.
- The `cached` field in the response tells the consumer whether tips are fresh or from cache.
- Different itineraries (even changing one activity) generate fresh tips.

### Prompt Design

The prompt asks Claude to respond with a strict JSON format and includes an example structure. This makes parsing reliable and predictable. The prompt also instructs Claude to give destination-specific tips rather than generic travel advice, which is more valuable for the traveller.

## What I'd Add Next

- **Input sanitisation**: Strip HTML/script tags from user input for extra security.
- **Rate limiting**: Prevent abuse with Laravel's built-in throttle middleware.
- **Logging**: Log each API call with Monolog (destination, response time, success/failure) for monitoring.
- **Configurable tip count**: Allow the client to request 1-5 tips instead of always 3.
- **Multi-language support**: Accept a `language` parameter and instruct Claude to respond in that language.
- **Database storage**: Store generated tips in a database table linked to itineraries for analytics and history.

## How I'd Extend This Into a Conversational Chatbot

The current endpoint generates tips in a single request — there is no memory or context between calls. To enable a real conversation where a traveller can ask follow-up questions about their itinerary, I would implement a **RAG (Retrieval-Augmented Generation)** approach:

### 1. Store Itinerary Data as Context

When an itinerary is created, store all its details (destination, dates, activities, hotel bookings, flight times, documents) in a structured database. This becomes the traveller's personal knowledge base.

### 2. Build a Conversation Endpoint

Create a new endpoint `POST /api/chat` that accepts:
```json
{
    "itinerary_id": "uuid-here",
    "message": "What time is my flight back?"
}
```

### 3. Retrieve Relevant Context (the "R" in RAG)

Before calling Claude, query the database for all information related to that itinerary: flights, hotels, activities, documents, and previous tips. This is the retrieval step — we find the specific data that Claude needs to answer accurately.

For more complex cases (many documents, long itineraries), a vector database like pgvector could be used to find the most relevant pieces of information based on semantic similarity to the user's question.

### 4. Augment the Prompt

Build the prompt by combining:
- A system message defining Claude as a travel assistant
- The retrieved itinerary context
- The conversation history (previous messages)
- The traveller's new question

This way Claude has all the information it needs to answer specific questions like "What restaurants are near my hotel?" or "Will it rain during my hiking day?"

### 5. Maintain Conversation History

Store each message (user + assistant) in a `conversations` table linked to the itinerary. Include the last N messages in each prompt so Claude maintains context across the conversation. This enables natural follow-ups like "What about vegetarian options?" after asking about restaurants.

### 6. Function Calling for Real-Time Data

Use Claude's function calling feature to let the chatbot fetch live information when needed — for example, calling a weather API when the traveller asks "Will it rain tomorrow in Tokyo?" or a maps API when they ask "How far is my hotel from Senso-ji?". The chatbot would decide which function to call based on the question.

This architecture would transform the current single-shot tip generator into a fully conversational travel assistant that knows everything about the traveller's specific trip.
